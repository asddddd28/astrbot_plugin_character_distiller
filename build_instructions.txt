# AstrBot 角色动态人格蒸馏器架构与构建流程

## 0. 项目定位

这个插件不建议做成一个简单的“角色设定生成器”，而应设计成一个 **角色动态人格蒸馏系统**。

它的目标不是把小说总结成几条“阳光、傲娇、冷静、热心”等性格词，而是从大量原文中构建一个可运行、可追溯、可阶段化、可检索、可持续修正的角色模型。

核心目标：

```text
原始文本
  ↓
高保真语料保存
  ↓
章节 / 场景 / 对话 / 心理描写 / 行为事件索引
  ↓
证据卡 Evidence Cards
  ↓
动态人格模型 Dynamic Persona Model
  ↓
AstrBot Persona / Knowledge Base / Memorix / Angel Memory / Self-Evolution
  ↓
运行时稳定扮演 + 记忆召回 + 人格弧线演化
```

系统最终应该能回答这些问题：

```text
这个角色在故事前期、中期、后期分别是什么状态？
这个角色在什么情况下会改变语气？
这个角色说话有什么固定句式和表达策略？
这个角色的表面语言和真实心理是否存在差异？
这个人格判断来自哪些原文证据？
有没有反例？
对话时应该调用哪些人格规则和原文样本？
运行过程中产生的新记忆是否会污染原著人格？
```

---

## 1. 总体架构

推荐插件名：

```text
astrbot_plugin_character_distiller
```

整体分为七层：

```text
┌─────────────────────────────────────┐
│ 1. Source Layer 原始文本层            │
│ 小说、剧本、设定集、聊天记录、角色语料 │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│ 2. Corpus Vault 高保真语料库          │
│ 原文、章节、段落、台词、场景、hash     │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│ 3. RAG / Retrieval 检索层             │
│ Knowledge Base、Memorix、SQLite FTS5  │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│ 4. Distillation 蒸馏层                │
│ 人格、语言、心理、关系、知识、反证     │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│ 5. Character Model 角色模型层         │
│ 人格卡、阶段人格、语言指纹、关系图谱   │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│ 6. Runtime Integration 运行时集成层   │
│ Persona、Self-Evolution、Memory 插件  │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│ 7. Evaluation & Review 评估审核层     │
│ 证据追踪、反跑偏测试、人工审核         │
└─────────────────────────────────────┘
```

一句话概括：

```text
Knowledge Base / RAG 负责找材料；
人格蒸馏器负责理解和建模；
Persona 负责常驻人格；
Memorix / Angel Memory 负责长期记忆；
Self-Evolution 负责运行时状态和人格弧线；
Corpus Vault 负责高保真原文追溯。
```

## 2. 各插件和系统的分工

### 2.1 AstrBot Knowledge Base

适合定位：**原文证据库 / 设定集检索库**。

推荐存放：

```text
小说章节
场景片段
角色对话样本
世界观设定
官方资料
角色访谈
```

不建议直接把整本小说作为一个大文件丢进去就结束。更好的做法是：

```text
按作品 → 阶段 → 章节 → 场景 切分成 md 文件；
每个文件头部加 metadata；
每个场景保留 chapter、phase、characters、scene、type 信息。
```

示例：

```markdown
# 第十二章 雨夜争执

metadata:
- work: xxx
- chapter: 12
- phase: middle
- characters: 角色A, 角色B
- scene: 雨夜争执
- type: dialogue+narration

## 正文
……
```

Knowledge Base 主要作用不是人格建模，而是让蒸馏器能稳定召回原文证据。

---

### 2.2 Memorix

适合定位：**长期记忆 + 图谱 + 可追溯记忆管理**。

推荐存放：

```text
蒸馏后的人格规则
角色阶段总结
角色关系记忆
用户与角色的互动记忆
事件时间线摘要
高置信知识卡
```

不建议让 Memorix 直接承担“从整本小说自动理解人格”的全部任务。它更适合作为蒸馏结果的长期记忆容器。

重要注意点：

```text
原著核心人格、世界观设定、关键关系应该设置为 protected / canon 类型；
运行时聊天记忆可以衰减、强化、冻结、恢复；
原著人格不能因为日常对话而被自动覆盖。
```

推荐标签：

```text
canon
persona
character:角色名
phase:early / middle / late / post_end
speech_style
relationship
worldbuilding
protected
```

---

### 2.3 Angel Memory

适合定位：**短条目核心记忆 / 主动记忆 / 角色认知层**。

推荐存放：

```text
100 字以内的人格短记忆
角色稳定行为规则
重要关系判断
世界观短知识
用户关系摘要
```

不推荐：

```text
整本小说
整章原文
超长 PDF
大段连续原文
几百万字未加工语料
```

Angel Memory 更适合保存“已经蒸馏好的短记忆”，例如：

```json
{
  "judgment": "角色A在中期阶段面对重要同伴受伤时，会用责备掩盖担心，但会立即采取实际照顾行为。",
  "reasoning": "多个中期场景显示其语言上责备、行动上保护。",
  "tags": ["persona:角色A", "phase:middle", "behavior:责备式关心"],
  "strength": 90,
  "memory_type": "persona"
}
```

---

### 2.4 Self-Evolution

适合定位：**运行时人格模拟 + 人格弧线 + 主动社交 + 反思**。

推荐承担：

```text
当前心情
当前能量
当前社交需求
最近对话状态
人格阶段 progress
主动续聊
用户关系变化
日结反思
```

不建议承担：

```text
解析百万字小说
识别角色台词归属
构建原著证据链
自动重写原著核心人格
```

最佳用法：

```text
蒸馏器输出 timeline_persona.json；
适配器把 timeline_persona.json 转成 Persona Arc Profile；
Self-Evolution 负责运行时阶段推进和当前状态注入。
```

关键原则：

```text
原著人格不可自动改写；
运行时情绪和关系可以演化；
人格补丁必须有证据和审核。
```

---

### 2.5 AstrBot Persona

适合定位：**常驻人格基底**。

推荐写入：

```text
角色身份
核心人格机制
语言风格规则
行为边界
反跑偏规则
基础 begin_dialogs
```

不推荐写入：

```text
整本故事情节
大量原文引用
所有事件记忆
所有关系细节
大量知识条目
```

Persona 应该短而稳定，记忆细节通过 RAG / Memorix / Angel Memory 按需召回。

## 5. 构建流程

### 阶段 1：文本导入

命令示例：

```text
/distill import --title 作品名 --file novel.txt
```

执行内容：

```text
1. 保存原始文件 original.txt。
2. 计算 sha256，保证可追溯。
3. 生成 normalized.txt。
4. 统一编码为 UTF-8。
5. 标准化换行、章节标题、空白字符。
6. 写入 source_meta.json。
```

注意事项：

```text
original.txt 永远不要覆盖；
所有二次处理都基于 normalized.txt；
每次重新导入都要记录 hash 和版本；
不要把原始文件存到插件代码目录。
```

---

### 阶段 2：结构切分

命令示例：

```text
/distill split work_001
```

切分层级：

```text
作品 Work
  └── 章节 Chapter
        └── 场景 Scene
              └── 段落 Paragraph
                    └── 台词 Utterance
```

优先级：

```text
1. 先按章节标题切。
2. 再按场景转换切。
3. 再按自然段切。
4. 最后识别台词。
```

场景切分依据：

```text
地点变化
时间变化
参与人物变化
事件目标变化
叙事节奏变化
```

---

### 阶段 3：角色识别和台词归属

命令示例：

```text
/distill characters detect work_001
/distill speaker resolve work_001 --character 角色A
```

需要处理：

```text
角色别名
昵称
称呼变化
旁白中的代词
多人对话中的说话人归属
心理描写归属
```

注意事项：

```text
台词归属不要完全相信模型；
应该保存 confidence；
低置信台词进入人工审核；
关键角色的语言风格样本必须人工抽查。
```

---

### 阶段 4：RAG 数据生成

命令示例：

```text
/distill rag export work_001 --target kb
/distill rag export work_001 --target memorix
```

Knowledge Base 推荐输入格式：

```markdown
# evidence_id: ev_work001_ch012_s03_001

metadata:
- work: 作品名
- chapter: 12
- phase: middle
- characters: 角色A, 角色B
- speaker: 角色A
- listener: 角色B
- emotion: 担心但掩饰
- speech_act: 责备式关心
- type: dialogue+narration

正文片段：
……
```

Memorix 推荐输入：

```json
{
  "text": "角色A在中期阶段面对重要同伴受伤时，会用责备掩盖担心，但会采取实际照顾行为。",
  "knowledge_type": "structured",
  "tags": ["canon", "persona", "character:角色A", "phase:middle", "protected"]
}
```

Angel Memory 推荐输入：

```json
{
  "judgment": "角色A在中期阶段面对重要同伴受伤时，会用责备掩盖担心。",
  "reasoning": "多处中期场景支持该模式。",
  "tags": ["persona:角色A", "phase:middle", "canon"],
  "strength": 90,
  "memory_type": "persona"
}
```

---

### 阶段 5：证据卡抽取

命令示例：

```text
/distill evidence build --character 角色A --work work_001
```

抽取维度：

```text
行为模式
语言模式
心理机制
价值观
目标和恐惧
关系变化
阶段变化
反跑偏证据
世界观知识
```

每张证据卡都必须包含：

```text
evidence_id
角色名
阶段 phase
结论 claim
原文位置
上下文范围
置信度
反证候选
```

注意事项：

```text
不要直接从单段文本生成强人格结论；
一个人格规则至少需要多个证据支持；
孤例只能作为候选，不应进入核心 persona；
必须保存反证。
```

---

### 阶段 6：人格蒸馏

命令示例：

```text
/distill persona build --character 角色A --phase auto
```

内部子任务：

```text
1. 人格机制蒸馏
2. 语言风格蒸馏
3. 心理活动蒸馏
4. 行为规则蒸馏
5. 关系模式蒸馏
6. 阶段人格蒸馏
7. 反跑偏规则蒸馏
```

不要输出这种低精度结果：

```text
角色A：阳光、自信、热心、温柔。
```

应该输出：

```text
角色A在公开场合倾向于用轻松语气稳定他人情绪；
但在独处或失败导致他人受伤后，会出现强烈自责；
他不直接表达恐惧，而是转化为行动承诺。
```

---

### 阶段 7：语言风格蒸馏

命令示例：

```text
/distill style build --character 角色A
```

应抽取：

```text
句长
语气
称呼习惯
口癖
标点习惯
反问频率
讽刺方式
情绪泄露方式
亲密表达方式
回避问题方式
不同关系下的语言差异
不同阶段下的语言差异
```

关键输出：

```text
speech_fingerprint.json
```

注意事项：

```text
语言风格不能只靠几句经典台词；
需要不同情绪、不同关系、不同阶段的样本；
不要保存大量原文到 prompt；
可以保存抽象句式和表达策略。
```

### 阶段 8：阶段人格建模

命令示例：

```text
/distill timeline build --character 角色A
```

阶段可以自动或手动：

```text

middle：中期
late：后期
post_end：结局后
custom：用户自定义章节范围
```

每个阶段应包括：

```text
身份状态
主导情绪
核心矛盾
关系模式
语言风格
行为规则
禁用表达
典型内心活动
阶段转移条件
```

注意事项：

```text
不能把前期防备和后期坦率混成一个人格；
需要支持用户锁定阶段；
默认运行时应选择一个明确阶段，而不是全阶段混用。
```

## 9. Skill 设计

### 9.1 persona-distiller/SKILL.md

职责：

```text
从证据卡中构建角色人格，不直接从原文片段随意总结。
```

核心规则：

```text
1. 不允许只输出性格标签。
2. 每个人格判断必须包含触发条件、表达方式、行为倾向。
3. 必须区分表层语言和真实心理。
4. 必须区分阶段人格。
5. 必须给出反跑偏规则。
6. 必须引用 evidence_id。
```

---

### 9.2 style-fingerprint/SKILL.md

职责：

```text
提取角色语言指纹。
```

核心规则：

```text
1. 分析句长、语气、称呼、口癖、标点、反问、讽刺、情绪泄露。
2. 区分不同对象、不同阶段、不同情绪下的语言变化。
3. 不要直接复制大量原文台词。
4. 输出抽象表达策略。
```

---

### 9.3 knowledge-distiller/SKILL.md

职责：

```text
从设定和剧情中提取短知识卡。
```

核心规则：

```text
1. 每条知识尽量短。
2. 保留来源 evidence_id。
3. 区分事实、推测、传闻、角色视角。
4. 不把剧情摘要误当世界观规则。
```

---

### 9.4 evidence-reviewer/SKILL.md

职责：

```text
审核人格判断是否有足够证据。
```

核心规则：

```text
1. 检查证据数量。
2. 检查是否存在反证。
3. 检查是否把短期情绪误判为长期人格。
4. 检查是否跨阶段混淆。
5. 给出 approve / reject / needs_more_evidence。
```

## 10. 运行时调用流程

最终对话时不要简单使用：

```text

```

建议使用：

```text

  ↓

  ↓

  ↓
```

```json
}
```

### 11.2 begin_dialogs 生成注意事项

begin_dialogs 应该成对出现，避免格式异常。

推荐格式：

```json

  "角色A：少自作多情。我只是觉得你再这样乱来，会给别人添麻烦。",
  "用户：谢谢你帮我。",
```

注意：

```text
1. 必须是偶数条，一问一答。
2. 不要放过长剧情。
3. 用来展示风格，不要承载全部设定。
4. 不要直接照抄大量原文。
```

## 12. 构建插件的注意事项

### 12.1 不要把人格蒸馏做成一次性总结

错误做法：

```text
把全文丢给 LLM，让它总结角色性格。
```

正确做法：

```text
切片 → 证据卡 → 多证据合并 → 反证检查 → 阶段建模 → 人格卡。
```

---

### 12.2 原文和记忆要分开

```text
原文：Corpus Vault / Knowledge Base
证据：Evidence Cards
人格：Persona Claims
短记忆：Memorix / Angel Memory
运行时状态：Self-Evolution
```

不要把整本小说全部写入 Angel Memory 或 Persona。

---

### 12.3 必须有 evidence_id

每条人格规则必须能追溯到原文。

```text
没有 evidence_id 的人格规则只能作为候选；
不能直接写入核心 persona。
```

---

### 12.4 必须支持反证

例如：

```text
角色A通常冷淡，但在第80章以后明显变得坦率。
```

这不是矛盾，而是阶段变化。

如果没有阶段建模，系统会误判成“人格不稳定”。

---

### 12.5 语言风格要和人格分开

人格解决“为什么这样说”；
语言风格解决“具体怎么说”。

两者不能混在一个字段里。

---

### 12.6 要防止角色模板化

不要输出：

```text
傲娇、温柔、腹黑、阳光、病娇。
```

应输出：

```text
触发条件 + 表层表达 + 内在动机 + 行为选择 + 禁止跑偏。
```

---

### 12.7 要防止 RAG 污染人格

RAG 召回可能出现：

```text
错误角色台词
错误阶段片段
非目标人物心理描写
同名角色混淆
世界观知识干扰人格判断
```

所以每次召回后都要做：

```text
角色过滤
阶段过滤
说话人过滤
证据类型过滤
置信度过滤
```

---

### 12.8 大文本处理要异步和可恢复

百万字文本蒸馏可能很慢，任务系统必须支持：

```text
任务状态保存
断点续跑
失败重试
分批处理
日志查看
取消任务
阶段性导出
```

---

### 12.9 版权和隐私注意事项

```text
不要把版权小说原文打包进插件仓库；
不要公开分发用户上传原文；
导出人格时尽量保存抽象规则，而不是大量原文；
用户上传的私有聊天记录要单独标记并允许删除。
```

---

### 12.10 Prompt 长度控制

常驻 Persona 不应无限变长。

推荐：

```text
核心人格 prompt：1000~3000 字
阶段人格补丁：500~1500 字
运行时记忆注入：3~10 条
原文样本召回：1~5 条
```

超过这个范围，模型容易被噪声干扰。

## 15. 最终原则

这个插件的核心原则可以总结为：

```text
1. 原文高保真保存，不直接塞进人格。
2. RAG 只负责找证据，不负责定义人格。
3. 人格必须由多证据合并得到。
4. 每条人格规则必须可追溯。
5. 角色必须分阶段建模。
6. 语言风格必须单独蒸馏。
7. 原著人格和运行时记忆必须隔离。
8. Self-Evolution 只能演化运行时状态，不能自动覆盖原著核心人格。
9. Angel Memory 和 Memorix 只保存结构化短记忆。
10. Persona Prompt 要短、稳、强约束。
```

最终目标不是“写一个很长的人设 prompt”，而是构建一个完整的角色运行系统：

```text
角色 = 原文证据
     + 阶段人格
     + 语言指纹
     + 内心机制
     + 关系图谱
     + 世界观知识
     + 运行时状态
     + 长期记忆
     + 反跑偏约束
```

这样做出来的人格蒸馏器，才不会只是“像角色介绍”，而是能在长期对话中尽量保持角色的语言、心理、关系和成长逻辑。
